## 2026-07-03 M5-export 実機検証の代表性不足（写真0枚のみで往復確認） [BAD]

- 事象: 「export→import一巡の実機確認済み」としたが検証レシピが写真0枚だったため、写真ありで必ず失敗するDexie tx罠（tx内で`fetch(dataUrl)`をawait→txが自動失効→"Transaction has already completed or failed"）を見逃し、ユーザー実機（写真2枚・スマホ→PC往復）で発覚
- 原因: ①実機検証データが機能の主要分岐（写真あり/なし）を代表していなかった ②レビューR1が同箇所のPromise.allを「メモリ観点」で指摘済み・R2でtx罠パターンをduplicateRecipeへ適用していながら、同一パターンの横断適用（importRecipe本体）をしなかった ③ユニットテストは`dataUrlToBlob`を即時resolveスタブで注入していたため罠が現れなかった
- 一般化ルール (次ループの指示文としてそのまま使える形で): 実機検証シナリオには機能の分岐を変えるデータバリエーション（写真あり/なし・0件/複数件）を必ず含める。レビューで危険パターンを1箇所指摘したら同一パターンをGrepで横断適用する。依存注入でスタブする外部APIには「実装と同じ非同期タイミング（マクロタスク遅延）」のスタブを最低1ケース入れる
- 反映先: **CLAUDE.md 昇格済み**（「実機検証の規律」— 検証の忠実度不足entryと同一根本原因〔検証が実利用を代表していない〕の2回目）
- 発生回数: 1 回目（同根の「検証の忠実度不足」と合算で2回目）

## 2026-07-03 M5-export レビューと実機検証の相互補完 [GOOD]

- 事象: UI層レビュー（R1）が見逃した「Home通常状態のImportJsonButton欠落」を、セッション実機検証のデザイン決定稿突き合わせが検出した。逆にレビューは実機で気づけないロジック欠陥（fetch失敗時の一括降格・Dexie tx罠）を検出し、2経路が補完し合った
- 原因: diffベースの静的レビューは「差分に現れない欠落（存在すべきなのに無いもの）」の検出が構造的に苦手。実機検証は画面単位で決定稿の要素実在を見るため欠落に強い
- 一般化ルール (次ループの指示文としてそのまま使える形で): UI層の出口では、レビュー委譲と並行してセッションが「決定稿にあるべきUI要素の実在確認」を実機で行う。レビュー委譲プロンプトにも「差分外も含め決定稿との突き合わせ再点検」を1観点として明示する
- 反映先: loop prompt
- 発生回数: 1 回目

## 2026-07-03 M5-export preview環境のCSSアニメーション停止 [BAD]

- 事象: ボトムシートが「開かない」ように見えた（translateY(100%)のまま座標検証が全滅）。実バグではなく、previewタブが `document.hidden=true` のときブラウザがCSSアニメーションのタイムラインを進めない環境制約だった（`getAnimations()`で `playState: running / currentTime: 0` を確認）
- 原因: 非表示タブではレンダリングパイプラインが止まり、`animation:`起点の開閉UIは初期キーフレーム位置に留まる。fill-mode未指定だと「閉じ位置で固まって見える」
- 一般化ルール (次ループの指示文としてそのまま使える形で): previewでアニメーション付きオーバーレイUIの座標検証を行う前に `el.getAnimations().forEach(a => a.finish())` で最終状態へ送る。「開かない・位置がおかしい」ように見えたら、コードを疑う前に `document.hidden` と `getAnimations()` の currentTime を確認する
- 反映先: loop prompt / CLAUDE.md 昇格候補（検証の忠実度不足entryと合わせてpreview検証の落とし穴が2件目）
- 発生回数: 1 回目

## 2026-07-03 mobile-ux-feedback 検証の忠実度不足 [BAD]

- 事象: ボトムシートのpointer-events透過バグ（✕・ハンドルが全て背後に素通し）を、実機確認済みとして出荷してしまった。検証がJSの`el.click()`直呼びだったためヒットテストを素通りし、後にユーザー実機で発覚
- 原因: `el.click()`はブラウザのヒットテスト（どの要素がタップを受けるか）を経由しない。overlay系UIはハンドラが正しくても座標上で受け取れないバグを持ち得る
- 一般化ルール (次ループの指示文としてそのまま使える形で): 操作系UIの実機検証では`document.elementFromPoint(中心座標)===対象要素`のヒットテストを必ず併用する。preview_clickはDPR≠1の環境で座標がズレて`<html>`に着弾することがあるため、失敗時はelementFromPoint＋座標付きイベントdispatchで代替する
- 反映先: loop prompt → **CLAUDE.md 昇格済み**（「実機検証の規律」）
- 発生回数: 2 回目（2026-07-03 M5でも同根〔検証が実利用を代表していない〕のデータ代表性不足が発生 → 昇格）

## 2026-07-03 M6-print-sns リフトアップ修正がz-index序列を逆転（実機ヒットテストで捕捉） [BAD]

- 事象: T40レビューM1（fixed-in-transform）対応のリフトアップ修正が、ShareDialogをExportSheetのスタッキング文脈（z:200）の外へ出したことでz序列を逆転させ（sheetBackdrop 200 > ShareDialog backdrop 60）、モバイルで共有ダイアログの全タップがシートbackdropに吸われるリグレッションを生成。同一レビュアーのRound 2はpointer-events継承・transform残存まで検証したが、z序列の再検証はセッションの観点指定からも漏れた。出口実機検証の`document.elementFromPoint`ヒットテスト（既存規律）が捕捉し、backdrop z-index 60→300の1行glueで解消・全ヒットテスト通過を再確認

- 反映先: **CLAUDE.md 昇格済み**（「委譲・レビューの規律」— M3「レビュー修正がリグレッションを生成」と同一根本原因の2回目）

運用ルール:

## 2026-07-03 mobile-ux-feedback 検証の忠実度不足 [BAD]

- 事象: ボトムシートのpointer-events透過バグ（✕・ハンドルが全て背後に素通し）を、実機確認済みとして出荷してしまった。検証がJSの`el.click()`直呼びだったためヒットテストを素通りし、後にユーザー実機で発覚
- 原因: `el.click()`はブラウザのヒットテスト（どの要素がタップを受けるか）を経由しない。overlay系UIはハンドラが正しくても座標上で受け取れないバグを持ち得る
- 一般化ルール (次ループの指示文としてそのまま使える形で): 操作系UIの実機検証では`document.elementFromPoint(中心座標)===対象要素`のヒットテストを必ず併用する。preview_clickはDPR≠1の環境で座標がズレて`<html>`に着弾することがあるため、失敗時はelementFromPoint＋座標付きイベントdispatchで代替する
- 反映先: loop prompt / CLAUDE.md 昇格候補
- 発生回数: 1 回目

## 2026-07-03 mobile-ux-feedback 並行委譲中のgit操作 [BAD]

- 事象: implエージェントが並行作業中に`git stash`を実行（直後にpopで復元・実害なし）。別エージェントの未コミット成果物を巻き込み消失させるリスクがあった。別のエージェントは禁止明記後に`git status`を1回実行し自己申告
- 原因: 委譲プロンプトの「git操作は不要」という表現が禁止として弱く、エージェントの自律判断でgitコマンドが混入した
- 一般化ルール (次ループの指示文としてそのまま使える形で): 並行委譲時のimplプロンプトには「git操作は禁止（stash/checkout/reset等の状態変更は絶対に行わない）」と禁止形で明記する。並行でない場合も成果物の管理はセッションの責務としgitはセッションのみが触る
- 反映先: loop prompt / CLAUDE.md 昇格候補
- 発生回数: 1 回目

## 2026-07-03 official-paint-charts [GOOD]

- 事象: ユーザー提供のカラーチャート（Citadel/Vallejo）を、Coat d'armsで確立した「docs/paints/*-master.json（原本）→ public/paints/*.json（配信・再生成）」の2層構成にそのまま載せられ、AK除外やレンジ付与も配信層の再生成だけで完結した
- 原因: 原本と配信を分離し、配信は常に原本から機械生成する構造にしていたため、データ差し替えが安全な単方向処理になった
- 一般化ルール (次ループの指示文としてそのまま使える形で): 外部由来データは「原本（docs・出典明記・無加工に近い形）」と「配信（アプリ最適形・原本から再生成）」の2層で持ち、配信側を直接手編集しない
- 反映先: loop prompt
- 発生回数: 1 回目

## 2026-07-02 M1-logic [GOOD]

- 事象: 独立タスク（T10/T11）の並列委譲が衝突なく機能し、レビューに「テストの実質性・境界値」観点を明示指定したところ境界値の独立再計算による検証まで実施された
- 原因: 並列委譲時に「相手エージェントの成果物ファイルへの不可侵」をプロンプトで明示／レビュー観点をマイルストーンの性質（ロジック層）に合わせて具体化した

- 反映先: loop prompt → **CLAUDE.md 昇格済み**（「委譲・レビューの規律」）
- 発生回数: 2 回目（2026-07-02 M2-dataでも同運用が機能: T12/T13・T14/T15の2回の並列委譲とデータ層観点レビュー）

## 2026-07-02 M2-data [GOOD]

## 2026-07-02 M3-presets レビュー修正がリグレッションを生成 [BAD]

- 事象: Round 1指摘への個別修正（key={colorId}化・palette再利用条件）が相互作用し、新規High 2件（チップ写真サイレント損失・ピッカー対話破壊）を生んだ。Round 2再レビューで捕捉
- 原因: 個別には妥当な最小修正を同一バッチで適用し、修正同士の複合作用の検証をimplのテストゲートだけに頼った
- 一般化ルール (次ループの指示文としてそのまま使える形で): レビューFAILの修正バッチ後は必ず同一レビュアーでRound N+1判定を取り、「修正同士の複合作用・修正が導入した新規問題」を明示観点に含める。UI状態管理（key・effect・確定タイミング）に触れる修正は特に要注意
- 反映先: loop prompt / CLAUDE.md 昇格候補
- 発生回数: 1 回目

## 2026-07-02 git初期化事故（セッション操作） [BAD]

- 事象: フォルダに既存の.git（ユーザーの手元コミット）があるのに新規初期化前提でgit操作し、checkoutで作業ツリーからファイルが一時的に消えた（既存コミットが全ファイルを保持していたため完全復旧、損失ゼロ）
- 原因: セッション開始時の環境情報「git repoではない」を、時間経過後（ユーザーが手元でinit/commit済み）も再検証せず使った
- 一般化ルール (次ループの指示文としてそのまま使える形で): 状態を変えるgit操作（init / checkout / branch -D / reset）の直前に必ず `git status && git log --oneline -3` で現状を確認する。セッション開始時のスナップショット情報は陳腐化する前提で扱う
- 反映先: CLAUDE.md 昇格候補
- 発生回数: 1 回目

## 2026-07-02 M4-editor impl委譲の外部要因切断（月次利用上限） [BAD]

- 事象: T25のimpl（Sonnet）が月次利用上限で切断され報告なしに停止（成果物はほぼ完成・テストのbeforeEach書き忘れと整形のみ欠落）。T27の初回委譲も着手前に切断（成果物ゼロ）
- 原因: 委譲先の失敗はプロンプト起因とは限らない。外部要因（利用上限・接続断）では「無報告のまま部分成果物が残る」状態になる
- 一般化ルール (次ループの指示文としてそのまま使える形で): impl委譲が報告なしに終了したら、まず `git status`＋全ゲート実行で残骸の完成度を判定する。①ほぼ完成→欠落のみセッションglue補完（切断点の痕跡=未使用importなどから欠落を特定）②成果物ゼロ→同一プロンプトで再委譲。中途半端な残骸の上への再委譲はしない
- 反映先: loop prompt → **CLAUDE.md 昇格済み**（「委譲・レビューの規律」に追記）
- 発生回数: 2 回目（2026-07-02 M4で2件／2026-07-03 公式チャート差し替えタスクで1件=npm実行中切断・成果物完全のためコミットのみで回収）

## 2026-07-02 M4-editor マイルストーン境界のタスク番号の谷間 [GOOD]

- 事象: M4入口の依存検算でT16（useRecipeStore）がM2完了(T12〜T15)とM3完了(T17〜T21)の番号の谷間で未実装のまま残っていたことを発見し、M4先頭で実装してブロッカーを回避した

- 一般化ルール (次ループの指示文としてそのまま使える形で): マイルストーン入口で「前マイルストーンまでの完了タスク番号が連続しているか」を検算し、谷間のタスクは着手前に依存グラフへ組み込む
- 反映先: loop prompt
- 発生回数: 1 回目
